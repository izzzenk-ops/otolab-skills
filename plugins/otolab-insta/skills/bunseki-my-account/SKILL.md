# 自分のアカウント分析（bunseki-my-account）｜裏側データ→次の一手が明確になるレポート

## このスキルについて

自分のInstagram投稿を、**アプリのインサイト裏側データ（外部リーチ・冒頭の視聴維持率・保存率・流入ソース）× 投稿の中身（冒頭ワード・台本構成）**で分析し、「次に何をやるか」が具体的に分かる1枚のHTML報告書を作る。

このスキルの主役は**アクション**。データはあくまで材料で、出力は「伸びなかった原因診断・冒頭文5案・テーマ提案・台本構成テンプレ・次に作る10本」という具体案。抽象的な所見で終わらせない。

**使い分け（重要）:**
- このスキル → **自分の**アカウントの投稿分析＋次の施策（裏側データを使う）
- `bunseki-competitor-account` → **競合・参考**アカウントの徹底分析
- `insta-concept-design` → 自分のアカウントのコンセプト設計そのもの

分析後、台本づくりは報告書の🎬ボタンから **daihon-copy-make**（や 〔ブランド名〕等の専用台本スキル）へ引き継げる。

## なぜ画面録画が要るのか（設計の背景）

外部リーチや保存率はWeb版インサイトからも取れるが、**「冒頭の視聴維持率カーブ」「平均視聴時間」「流入ソース内訳」はスマホアプリのインサイトにしか無い**（PC版・APIでは取得不可。検証済み）。そこで、ユーザーがアプリのインサイト画面をスクロール録画したものを入力にする。録画からフレームを抜き出してClaudeが数値を読む。

## 前提条件

**受講生に配って同じ状態で使える自己セットアップ型**（[[feedback-skills-student-distributable]]）。真の前提はこれだけ：

- **Mac（Apple Silicon）** または **Windows**（※Windows版はベータ・実機未検証）
- Chrome拡張（Claude in Chrome）が接続済みで、Chromeで自分の instagram.com にログイン済み
- Claude Code が使えること
- **自分のスマホでインサイトの画面録画が撮れること**（撮り方はSTEP 1で案内する）

`yt-dlp`・`ffmpeg`・文字起こしライブラリは入っていなくてよい。STEP 0のセットアップが自動で入れる。**文字起こしエンジンはOSで自動切替**（`transcribe_engine.py`）：Mac（Apple Silicon）=mlx_whisper／Windowsなど=faster-whisper。

## 出力先

```
~/Documents/Claude/Projects/インスタ自己分析/<username>_<YYYYMMDD>/
├── data/        insights.json（録画から読んだ裏側数値）, reels.json, selection.json, stats.json
├── frames/      録画から抽出したフレーム（<録画名>/f_NNN.jpg）＋ frames_manifest.json
├── reels/       動画mp4＋カバー画像jpg＋download_log.json
│   └── transcripts/   ID.txt（全文）, ID.segments.json
├── style_profile.md  台本スキル引き継ぎ用（自分の勝ち型）
└── report.html  最終報告書（画像は相対パス参照なのでフォルダごと移動可）
```

## デフォルトの分析対象

**伸びた投稿 5本前後 ＋ 伸びなかった投稿 3本前後 ＋ 最新数本**。ユーザーが本数や特定投稿を指定したらそれに従う。伸びた/伸びなかったの判定は、録画のアカウント「コンテンツ」タブの閲覧数一覧や、STEP 2で集める再生数の中央値（平常値）を基準にする。

## STEP 0：環境セットアップ＋ツール読み込み

**最初に必ず環境セットアップを実行する**（冪等なので毎回流してよい）。**実行OSを判定して対応するスクリプトを走らせる**：

```bash
# Mac（Apple Silicon）
bash <skill>/scripts/setup.sh
```
```powershell
# Windows（PowerShell）
powershell -ExecutionPolicy Bypass -File <skill>\scripts\setup.ps1
```

- 全項目が OK で「セットアップ完了」なら次へ。初回はツール導入で数分かかることがある（Mac=Homebrew/mlx_whisper、Windows=winget/faster-whisper）
- `!`（Windowsは`[!]`）が残る場合はその項目をユーザーに伝える（再実行、またはターミナル/PowerShellを開き直すと解決することが多い）

Chromeツールが遅延読み込みの場合、**1回のToolSearch**でまとめてロード：

```
select:mcp__claude-in-chrome__tabs_context_mcp,mcp__claude-in-chrome__navigate,mcp__claude-in-chrome__computer,mcp__claude-in-chrome__get_page_text,mcp__claude-in-chrome__javascript_tool
```

出力先ディレクトリを作成し、このスキルのscripts/があるパス（base directory基準）を控える。

## STEP 1：インサイト画面録画を受け取る／撮り方を案内する

ユーザーがすでに録画ファイル（.mp4/.MOV等）を渡していれば、そのパスを控えてSTEP 1bへ。まだ無ければ、**以下の撮影レシピをそのまま提示**して撮ってもらう（撮れたらMacにAirDrop、`~/Downloads` に入る）。

> **録画①：アカウント全体（約40秒）**
> プロフィール右上の三本線 → インサイト（プロアカウントのダッシュボード）→ 期間を「過去30日間」に。ゆっくりスクロールしながら、リーチ推移／エンゲージメント／**オーディエンス（地域・年齢・最もアクティブな時間帯）**／フォロワー増減で1〜2秒ずつ止まる。
>
> **録画②：伸びたリール 3〜5本（各20〜30秒）**
> 各リールを開く→インサイトを見る→最後までゆっくりスクロール。特に **リーチ内訳（フォロワー/以外）・平均視聴時間・視聴維持率グラフ（あれば3秒静止）・保存/いいね/シェア・流入ソース** で止まる。
>
> **録画③：伸びなかったリール 2〜3本（各20〜30秒）**
> 同様に。特に**維持率グラフ＝どこで離脱したか**を写す。
>
> 撮り方のコツ: 1画面ずつ止まってゆっくり。速すぎると読み取れない。古い投稿（目安2年以上前）は「インサイトデータがありません」と出ることがある＝直近の投稿を選ぶ。

### STEP 1b：録画からフレーム抽出＆数値の読み取り

```bash
python3 <skill>/scripts/extract_frames.py <出力先>/frames <録画1> <録画2> ...
```

- 各録画が `frames/<録画名>/f_NNN.jpg` に、画面が1.5秒ごとに切り出される（動画1本あたり数十枚）
- **抽出フレームを順にReadして数値を読み取り、`data/insights.json` に構造化して保存する**。読む対象と保存キーの目安：
  - **アカウント全体**（録画①）: フォロワー数・30日リーチ・フォロワー外比率・コンテンツタイプ別閲覧数（リール/投稿/ストーリーズ）・**最もアクティブな時間帯**・主要オーディエンス層・プロフィールアクセス・リンクタップ・閲覧数順の投稿ランキング
  - **各リール**（録画②③）: カバー画像で識別できる特徴／キャプション冒頭・閲覧数・リーチ・**フォロワー外比率**・平均視聴時間・**冒頭維持率（0〜3秒で何%まで落ちるか＝カーブの形）**・保存/いいね/コメント/シェア・**流入ソース内訳**（フィード/発見/リールタブ/ストーリーズ/プロフィール %）
- 全フレームを1枚ずつ丁寧に読むとトークンを消費する。**同じ画面が続くフレームは飛ばし、"数値が写っている代表フレーム"だけを読む**（manifestの順に眺め、変化したところを拾う）
- 数値が読めないフレームがあっても止まらない。読めた分だけ埋め、読めなかった項目は後で報告書に「録画に写っていないため不明」と正直に書く
- **各リールを、STEP 3でダウンロードするリール（URL）と対応づけられるよう**、insights.jsonの各リールに「カバーの特徴・キャプション冒頭・閲覧数」を必ず残す（後で文字起こしと突き合わせるための手がかり）

## STEP 2：自分のリール一覧を集める（Chrome）

`https://www.instagram.com/<自分のusername>/reels/` に移動（未ログインなら「Chromeで自分のIGにログインして」と伝えて終了）。**制約3つは競合スキルと同じ**：

1. navigate直後はグリッド未描画。**2〜3秒待ってから**収集
2. 再生数テキストはアクセシビリティラベル付き（「カウントアイコンを見る29.7万」）。ラベルを除去して数字だけ抽出
3. 署名付きCDN URL（サムネ）はツール結果に含めるとBLOCKED。**サムネURLは集めず、パス・再生数・ピン留めだけ**をJSONで返す（サムネはSTEP 3でyt-dlpが取る）

グリッドは仮想化されているので `javascript_tool` でwindowアキュムレータに溜めながらスクロール：

```js
window.__ig = window.__ig || {};
document.querySelectorAll('main a[href*="/reel/"]').forEach(a => {
  const pinned = !!a.querySelector('svg[aria-label*="固定"], svg[aria-label*="Pinned"]');
  const key = a.href.split('?')[0];
  const m = (a.textContent || '').replace(/カウントアイコンを見る/g, '').trim().match(/^[\d.,]+(万|億|千)?/);
  if (!window.__ig[key]) window.__ig[key] = {thumb: null, views: m ? m[0] : '', pinned};
});
window.scrollBy(0, window.innerHeight * 3);
Object.keys(window.__ig).length
```

- 件数が2回連続で増えなくなるまで繰り返す（1〜2秒間隔）。直近100本程度で打ち切ってよい
- `JSON.stringify(Object.fromEntries(Object.entries(window.__ig).map(([u,v]) => [new URL(u).pathname, v])))` で取り出し、`data/reels.json` に保存

選定と統計：

```bash
python3 <skill>/scripts/select_reels.py data/reels.json data/ --top 8 --latest 4
```

- `stats.json` の中央値＝平常値。伸びた/伸びなかったの線引きに使う
- **録画に写っていた投稿（insights.jsonのリール）は必ず選定に含める**（DL対象に手動追加してよい）。伸びなかった投稿の分析には「録画で維持率が写っている低再生の投稿」を選ぶ

## STEP 3＋4：自分のリールをDL＆文字起こし（並行パイプライン）

```bash
# Mac
~/telop-tool/venv/bin/python3 <skill>/scripts/pipeline.py data/selection.json reels/ --language ja
```
```powershell
# Windows
& "$HOME\telop-tool\venv\Scripts\python.exe" <skill>\scripts\pipeline.py data\selection.json reels\ --language ja
```

- 文字起こしエンジンはvenv内で自動切替（Mac=mlx_whisper／Windows=faster-whisper）。**Windowsのfaster-whisperはCPU動作なので、初回はモデルDL＋各本の処理にMacより時間がかかる**（ベータ）
- DL（2秒間隔・cookieなし→失敗時chrome cookie）と文字起こし（モデルのロード1回）を並行実行。済みファイルはスキップ
- **パイプライン実行中に、STEP 1b/2の情報で先に分析を書き進める**（診断マトリクス・全体サマリー・裏側数値の整理）。完了時に残るのは冒頭/構成分析と報告書組み立てだけになる
- 一部失敗しても続行。失敗分は報告書に正直に書く
- Whisper注意: 固有名詞の誤変換はあり得る（明らかなものは文脈補正、創作しない）。**末尾の同一文字繰り返し（幻覚）は無視**

## STEP 5：分析（このスキルの核心）

**実データだけを根拠にする。** 録画の数値・文字起こし・カバー画像を突き合わせ、以下を作る。ここが薄いとスキルの意味が無い ── 常に「具体的に・次にやること」に落とす。

**A. 中身×数値の突き合わせ** — 各リールについて、insights.jsonの裏側数値と、transcripts/の冒頭（segments.jsonの0〜3秒）・構成を紐づける。カバー特徴・閲覧数で対応をとる

**B. 診断マトリクス（リーチ×維持率の4象限）** — 分析対象の各投稿を4象限に配置：
- 高リーチ×高維持＝**勝ちパターン**（増やす・シリーズ化）
- 低リーチ×高維持＝中身◎届いてない（**作り直して再投稿・宣伝**）
- 高リーチ×低維持＝フック◎中身薄い（**展開・オチを直す**）
- 低リーチ×低維持＝**テーマから見直し**

**C. 伸びなかった原因診断** — コケた各投稿を1行で断定し、直し方をセットで。維持率カーブが冒頭で急落なら「フックが弱い」、リーチは出たが保存率が低いなら「テーマに価値提示が無い」、そもそもリーチが出ていないなら「テーマ/カバーで外に出ていない」等、**数値を根拠に切り分ける**

**D. 冒頭文の提案 ×5** — 外部リーチ・保存が伸びた投稿の冒頭を型に分類（自己開示/逆張り/数字/悩み提起/結論先出し等）し、平均で効いている型を特定。その型で**そのまま撮れる冒頭文を5案**作る（データ根拠を添える）

**E. テーマ提案** — 保存率・外部リーチが跳ねたテーマの共通項から、次に当てにいくテーマを3〜5個。どの投稿が根拠かを明記

**F. 台本構成テンプレ＋サムネの型** — 伸びた投稿の文字起こしから「自分の勝ち構成（フック→展開→オチ→CTA）」をテンプレ化。維持率が保てた尺、カバー（サムネ）の文字量・配色・構図の共通ルールも抽出

**G. 次に作る10本リスト** — A〜Fを統合し、テーマ×冒頭文×尺×サムネ方針をセットにした10本を、**今すぐ効く順**に並べる（そのまま撮れる状態にするのがゴール）

分析後、台本生成用の **`style_profile.md`** を出力フォルダ直下に保存する（**daihon-copy-makeがこれを読む**）。含める内容：一言コンセプト／自分の勝ちフックの型（実例つき）／展開のブロック構造／オチの型／文体ルール（人称・語尾・頻出フレーズ）／維持率が保てた尺と文字数／参考にすべき文字起こしファイル名2〜3本。書き方の見本は競合スキル側 `~/Documents/Claude/Projects/競合アカウント分析/kanako_vlog_20260710/style_profile.md`

## STEP 6：HTML報告書の生成

1. `<skill>/assets/report_template.html` を読み、**CSSと構造はそのまま**に中身を実データで埋めて `report.html` として保存（`[[...]]`を置換、カード/行/chipは必要数複製）
   - セクション③のリールカードには裏側数値バッジ（外部リーチ%・保存率・冒頭維持%）と流入ソースバー、**文字起こしの`tr-box`ブロック**（開閉＋コピー＋.txt保存）を付ける。全文はHTMLエスケープして`<pre class="tr-text">`へ。テンプレ末尾の`<script>`も残す
   - セクション②の4象限には各投稿を`<span class="chip">`で配置。⑧の10本リストは`plan-table`の行を10本複製
   - カード/行が多い場合は、Editの繰り返しよりPythonでtranscripts/*.txtを読んで一括注入する方が確実
2. 画像は相対パス（`reels/<ID>.jpg`）参照。DL失敗リールのカードは画像なし（枠のみ）でよい
3. フッターに「確認できなかった項目」を正直に記載（録画に写っていない数値・インサイトデータ無しの投稿・DL失敗など）
4. 完成したらChromeで開く（`file://`はChrome拡張のnavigateでは開けないので必ず`open`）：

```bash
open -a "Google Chrome" ~/Documents/Claude/Projects/インスタ自己分析/<dir>/report.html
```

5. 表示検証はヘッドレスChromeでスクリーンショットを撮り、画像・レイアウト崩れがないか確認：

```bash
"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu \
  --screenshot=<scratchpad>/report_check.png --window-size=1200,2400 --hide-scrollbars \
  "file://<dir>/report.html"
```

## 注意事項

- ダウンロード間隔（2〜3秒）・スクロール間隔を詰めない（レート制限・BAN回避）
- 収集した動画・画像・録画は自己分析目的のローカル保存
- 長時間処理（DL・文字起こし）は run_in_background で走らせ、待ち時間に分析を進める
- 途中で何かが失敗しても**報告書を最後まで届けることを最優先**。失敗は隠さず報告書とチャットの両方に書く
- 数値は「録画で読めた実データ」だけを使う。読めなかったものを推測で埋めない（切り分けの根拠が崩れる）
