# リール1本 徹底分析（bunseki-reel）｜1本を解剖→ビジュアル報告書→台本へ引き継ぎ

## このスキルについて

参考になるリール**1本**のURLを受け取り、**実データ（動画・サムネ・音声文字起こし・フレーム・コメント欄）に基づいて**その1本だけを縦に深掘りし、1枚のHTML報告書としてChromeで開く。運用中に「このリール上手いな」と思ったとき、フック・構成・CTAの設計を丸ごと解剖して**自分のリールに移植する**ためのスキル。

**使い分け（重要）:**
- このスキル → **リール1本**を徹底解剖。`instagram.com/reel/<id>` が1本貼られたとき
- `bunseki-competitor-account` → **アカウント丸ごと**（プロフィール＋上位/最新リール横断）
- `bunseki-my-account` → 自分のアカウント分析
- `instagram-analysis` → 閲覧ベースの軽量レポートをDiscordへ

分析結果はそのまま **daihon-copy-make**（型移植台本スキル）に引き継げる（報告書の🎬ボタン、詳細はSTEP 7）。

## 前提条件

**受講生に配って同じ状態で使える自己セットアップ型**（[[feedback-skills-student-distributable]]）。**Mac / Windows 両対応**。真の前提はこれだけ：

- **Mac（Apple Silicon）または Windows**（Linuxも可）。Intel Macも可
- **Python 3.9+ が入っていること**（Macは通常 `python3` あり。Windowsは python.org でインストール時「Add to PATH」にチェック）
- Chrome拡張（Claude in Chrome）が接続済みで、Chromeで instagram.com にログイン済み
- Claude Code が使えること

`yt-dlp`・`ffmpeg`・文字起こしエンジンは**入っていなくてよい**。STEP 0の `setup.py` がpipだけで自動セットアップする（Homebrew等のシステムパッケージマネージャ不要。ffmpegは imageio-ffmpeg 同梱バイナリを使う）。文字起こしエンジンはOSで自動出し分け：**Apple Silicon Mac=mlx_whisper（Metal・高速）／それ以外=faster-whisper（CPU/GPU）**。

その他：
- 所要時間の目安: **3〜6分**（1本なので短い）。**初回のみ**環境構築＋文字起こしモデルのダウンロードで数分〜十数分（特にWindowsのfaster-whisperは初回モデルDL〜数GB・CPU推論はmlxより遅い）。開始時にユーザーへ目安を伝える
- コスト**0円**。文字起こし・分析・台本引き継ぎはすべてローカル＋Claude Code内で完結する（API課金なし）

## STEP 0：環境セットアップ＋ツール読み込み

冪等なので毎回流してよい。**OSでコマンドが違う**（Pythonの呼び名が違うだけ）：

```bash
# Mac / Linux
python3 <skill>/scripts/setup.py
```
```bat
REM Windows（PowerShell / cmd）
python <skill>\scripts\setup.py
```

- 最後に「✅ セットアップ完了。以下の python で実行してください: `<venv-python>`」が出たらそのパスを控える。以降のスクリプトは**必ずこの venv python** で実行する：
  - Mac/Linux: `~/.bunseki-tools/venv/bin/python`
  - Windows: `%USERPROFILE%\.bunseki-tools\venv\Scripts\python.exe`
- 初回はエンジン（mlx/faster-whisper）のインストールで数分。`⚠` が出たらエラーをユーザーに伝える（Pythonが無い場合が多い→前提条件を案内）
- **どのOSかは `setup.py` の出力（`OS: ...`）で分かる**。以降のコマンドはそのOSに合わせて venv python のパスを組み立てる

Chromeツールが遅延読み込みなら**1回のToolSearch**でまとめてロード：

```
select:mcp__claude-in-chrome__tabs_context_mcp,mcp__claude-in-chrome__navigate,mcp__claude-in-chrome__computer,mcp__claude-in-chrome__get_page_text
```

出力先ディレクトリ名を決める（ユーザー名とリールIDは STEP 1 のメタで確定するので、仮に `reel_<id>_<日付>` で作って後でrenameしてもよい）。scripts/ のパス（base directory基準）を控える。

## STEP 1：動画・メタ・文字起こしの取得

リールURLから**1本を取り込む**（yt-dlpでDL＋メタ、文字起こしまで一気に）。`<venv-python>` は STEP 0 で控えたパス：

```bash
<venv-python> <skill>/scripts/fetch_reel.py "<リールURL>" "<out_dir>" --language ja
```

- yt-dlpはcookieなし→失敗時のみ `--cookies-from-browser chrome` で自動再試行。公開リールは通常cookieなしで落ちる
- `data/meta.json` に いいね数・コメント数・再生数・尺・キャプション・投稿日・uploaderが入る。**再生数はIG web経由だと `null`（非表示）のことが多い**。その場合は報告書に「再生数非表示」と正直に書く
- **文字起こしはバックエンド自動切替**（`transcribe_core.py`）。Apple Silicon Mac=mlxで重複ウィンドウ方式、それ以外=faster-whisperでVAD方式。どちらも**冒頭がBGMのみ・途中に無音でも中盤を落とさない**（単発方式は冒頭BGMのリールで中盤を丸ごと欠落させる事故があるため、この設計にしている）
- 出力を確認：`data/transcript.txt` と `data/meta.json` を読む。uploaderのハンドルでディレクトリ名を確定（必要ならrename）

**文字起こしの注意（最重要・過去の失敗）:**
- **壊れた文字起こしを根拠に結論を書かない。** かつて、冒頭がBGMのみのリールで文字起こしが空/幻覚（「Buzzleのリンク」の繰り返し）になったのを見て「ナレーション無しのテロップ型」と即断し、実際は途中から入っていたナレーションを見落とす失敗をした。**二度と繰り返さない。**
- `fetch_reel.py` が `meta.json` に `transcript_warning` を書いていたら（空・セグメントが尺超過・幻覚ループのいずれか）、**STEP 2のフレームで人物の口の動き・字幕を必ず確認する**。喋っているのに文字起こしが空/変なら、それは「ナレーション無し」ではなく**文字起こしの失敗**。動画を消して `fetch_reel.py` を再実行する（全区間を舐める設計なので通常はこれで解決する）
- 「ナレーション無し（BGMのみ）」と断定してよいのは、**フレームで口が動いていないことを確認できたときだけ**。文字起こしが空という事実だけで断定しない
- 窓の重なりで**軽微な重複や言い直し**が残ることがある（例: 同じ台詞が2回、断片「自分よ」等）。分析時に文脈で1本の台本に整える
- Whisperの**固有名詞・同音異義の誤変換**はあり得る（例: 「宣伝」→「洗練」）。フレームのテロップと突き合わせて明らかな誤りは補正する。**創作はしない**

## STEP 2：フレーム抽出とテロップ・構成の解析

**リールは音声だけでは分析できない。** テロップ・画・サムネ帯・画面録画などオンスクリーンの情報が構成の半分を占める。フレームを抜いて必ず目で見る：

```bash
<venv-python> <skill>/scripts/extract_frames.py "<out_dir>/<リールID>.mp4" "<out_dir>/frames" --count 12 --width 360
```

- ファイル名にタイムスタンプが入る（`f_03.5s.jpg` = 3.5秒地点）。**Readで全フレームを見て**、各時点の「画・テロップ・演出」を起こす
- 文字起こし（音声）と突き合わせて、**秒ごとの構成マップ**を組み立てる：どの秒に何を見せ、何を言い、いつテロップを出し、いつ転換し、どこでCTAへ行くか
- **数字の出どころを区別する**：動画の中に出てくる再生数（例「5700万再生」）は、多くの場合**引用元動画や実績の提示**であって、この投稿自身の再生数ではない。混同しないこと
- 尺が長い/テロップが速い場合は `--count 16` 等に増やす

## STEP 3：Chromeでキャプション・コメント欄を収集

`fetch_reel` のメタで大半は取れているが、**コメント欄の反応**はChromeでしか取れない。ここがこのスキルの独自価値：

1. `navigate` でリールURLへ。数秒待ってスクショで状態確認
   - 未ログイン → 「Chromeでinstagram.comにログインしてから再度」と伝えて終了
   - 非公開/削除 → その旨伝えて、取れた範囲（動画・メタ）だけで続行
2. キャプションの「…さらに表示」を展開し、`get_page_text` で全文を確認（meta.jsonのcaptionと突き合わせ）
3. **コメントアイコンをクリックしてコメントパネルを開く**。スクショ＋`get_page_text`で上位コメントを読む。見るポイント：
   - **CTAワードの書き込み**：キャプションが「〇〇とコメント」で誘導している場合、コメント欄がそのワードで埋まっていれば**コメント誘導→自動DMのファネルが機能している証拠**（各コメントへの返信有無も見る）
   - 質問／称賛／批判の比率、視聴者が刺さったポイント
4. 再生数がUI上に見えれば控える（`meta.json`がnullでもここで補完できることがある）

1. 次を順に探す：

   - `~/Documents/Claude/Projects/インスタコンセプト設計/` 配下の最新のコンセプト/プロフィール成果物（insta-concept-designの出力）

2. **見つかれば** 読み込み、STEP 4-⑧の活かし方をその発信に合わせて個別化する
3. **見つからなければ** 活かし方は汎用のまま書き、STEP 7で「発信プロフィールを作るか」を提案する（下記）

## STEP 4：分析

実データ（動画・フレーム・文字起こし・メタ・コメント）だけを根拠にする。見ていないものを「〜のはず」と書かない。**8つの観点**で解剖する：

1. **数字とポジション** — いいね・コメント（取れれば保存/再生）。この投稿の**目的**は何か（認知/保存/コメント誘導→DM/販売導線）。コメント率などの比率も
2. **冒頭フック＆サムネ設計** — 最初の2〜3秒の「画」と「音」。なぜ指が止まるか（違和感/数字/強い動き/断定）。固定サムネ帯の文字数・色・強調・配置。引きの作り方（「〜した結果」で結論を先送り等）
3. **構成マップ（秒配分）** — segments.jsonとフレームから起こした時系列の設計図。フックが尺の何%か、どこで転換し、どこでCTAへ行くか。**各ブロックに「視聴者心理（推定）」を1行添える**（下記4の考え方で）
4. **視聴者心理の設計図（伏線・回収・感情の流れ）** ⭐ — この1本の"戦略"を言語化する新しい柱（[[reference-viewer-psychology-blueprint]]）。**制作者の内心は断定せず、「視聴者に起きる効果」を軸にする**：
   - 各瞬間で視聴者は何を**感じ・考え・次にどうする**か
   - **伏線→回収**のマップ：冒頭で何を仕込み（未回収の疑問・違和感）、どこで回収したか
   - **感情の動き**：引き→緊張→解放／期待→裏切り→納得 などの弧
   - **CTA直前の心理状態**：どういう気持ちにさせてからコメント/フォローに誘導しているか
   - ⚠️ **必ず「推定」と明示**する。断定してよいのは視聴者側の反応まで（他人の頭の中は書かない）

   **⭐️具体性ルール（最重要）— 抽象語で書かない。** 「疑問を仕込む」「自分ごと化」「センシティブな引き」だけでは、この報告書を見たユーザーが**自分の台本に転換できない**。必ず次の4点で、**その動画の具体ワードを引用して**書く：
   1. **具体トリガー**（画面/言葉の"何が"）
   2. **視聴者の頭に浮かぶ実際の言葉**「〜」— **主語・目的語を必ず入れる**。曖昧な指示語で終わらせない（❌「こんなに溜まってるの？」→ ✅「こんな**洗い物**が溜まってるの？」）
   3. **意図動詞で締める** — 生の言葉を宙ぶらりんにせず「〜という**疑問を抱かせる**／**と思わせる**／**したくさせる**」と、何を狙った設計かを明示する
   4. **それが次に何をさせるか**（見続ける/コメントする/欲しくなる 等）

   基準の解像度は制作者本人の自己分析レベル（例:「冒頭に山盛りの汚れた食器を"説明せず"映す→"なんでこんな洗い物が溜まってるの？"という疑問を抱かせる→答えを知りたくて最後まで見る」）。ユーザーが自分の題材にそのまま置き換えて台本にできる粒度まで落とす。
   - ❌「冒頭で疑問を仕込む」→ ✅「冒頭に◯◯を説明せず映す→"なんでこの◯◯が△△なの？"という疑問を抱かせる→答えを知りたくて見続ける」
   - ❌「センシティブな引き」→ ✅「"妊娠かも？"と際どい話題で気を引く→"でも違うんだ"と思わせる→検査薬の展開で引っ張る→豆知識をあえて言い切らずコメントに余白を残す」
5. **台本・テロップ全文と型** — 音声＋テロップを1本に整えた全文。フック型／展開構造／オチ・CTA型／口調・語尾・頻出フレーズ／尺と文字数。**この1本の「型」をテンプレート化**する（daihon-copy-makeが読む形）
6. **演出・CTA・収益導線** — テロップの出し方・強調、画の作り方（素材/顔出し/画面録画の組み合わせ）、最後の誘導先（コメント誘導→自動DM/リンク/フォロー）の仕組み
7. **コメント欄の反応** — 実際に何が書かれているか。CTAが機能しているか。**観点4の「狙った効果」が実際に起きているかの答え合わせ**にもなる（例: 疑問を仕込んだ→コメントで質問が来ている）
8. **活かし方** — 明日から真似できる具体アクション4〜6個。各アクションは「この動画の◯◯（具体）→ 自分の題材ならこう置き換える」の形で書く。**フック／サムネ文字／構成の秒配分／視聴者心理の設計（どこで何を感じさせるか）／CTAの型／演出**をカバーする。**丸パクリではなく構造と心理設計の移植**を勧める
   - **個別化の材料は STEP 3.5 で確認した「発信プロフィール」だけを使う**（メモリを勝手にあさらない）：
     - プロフィールが**あれば** → その発信（誰に／何を／目的）に合わせて活かし方を具体化する（報告書見出し下に「◯◯さんの発信向けに具体化」と一言添えてよい）
     - **無ければ** → 特定アカウントに寄せず**汎用**で書く（見出し下は「自分の題材に置き換えて使う具体アクション」）。⚠️ メモリにあるけんじ個人や他人のアカウント名（〔ブランド名〕／〔アカウント例〕／〔アカウント例〕 等）を勝手に報告書へ書かない（受講生配布で他人の情報が混ざるため）。この場合 STEP 7 でプロフィール作成を提案する

## STEP 5：style_profile.md（daihon-copy-makeへの引き継ぎ）

分析が終わったら、台本生成用の **`style_profile.md`** を出力フォルダ直下に保存する（daihon-copy-makeがこれを読んでユーザーの台本を作る）。**1本の型なので断定しすぎない**。含める内容：

- 一言コンセプト（この1本が何をする動画か）
- フックの型（実例つき）
- 展開のブロック構造（秒配分つき）
- オチ／CTAの型
- **視聴者心理の設計（心理レバー）** ⭐ — 各ブロックで視聴者に起こす効果（推定）、仕込んだ伏線→回収、感情の弧、使われている心理レバー、CTA直前の心理状態。**daihon-copy-makeが台本を書くとき、言葉より先にこの"効果"を設計するための材料**。効果ベースで書く（制作者の内心は書かない）。**必ず具体トリガー＋視聴者の言葉「〜」＋次の行動 の3点セットで、自分の台本に移植できる粒度で書く（抽象語で止めない）**
- 文体ルール（人称・語尾・頻出フレーズ）
- 尺と文字数
- 移植時の注意（固有の題材・ギャグ・固有名詞の丸写し禁止）

機械可読なプレーンMarkdownで書く（報告書セクション4・5と同じ内容）。書き方の見本は競合分析側 `~/Documents/Claude/Projects/競合アカウント分析/〔参考アカウント〕_20260710/style_profile.md` を参考にしてよい。

## STEP 6：HTML報告書の生成

1. `<skill>/assets/report_template.html` を読み、**CSSと構造はそのまま**に `[[...]]` を実データで置換して `report.html` として保存
   - ⚠️ **本文に内部・技術用語を一切出さない。** 報告書を読むのは受講生／運用者であって、処理の裏側には関心がない。次のような語は本文に書かない → `Whisper`／`segments.json`／`フレーム`（→「最初の場面」「画面」等）／`mlx`／`faster-whisper`／`yt-dlp`／`重複ウィンドウ`／`meta.json`／`transcript`／`IG web`／`幻覚`／`skill名`。すべて中学生にも伝わる平易な言葉に言い換える（文字起こしの補正も「聞き取りにくい箇所は前後から補いました」程度に）
   - **文字起こし全文は必ず`tr-box`に埋め込む**（HTMLエスケープし、句点「。」ごとに改行を入れて読みやすく。末尾の同一文字繰り返し等の誤りは除去）。台本スキルにワンクリックで渡せるのが価値

   - 画像は相対パス（`<ID>.jpg`＝カバー, `frames/f_xx.xs.jpg`＝フレーム）で参照。動画・カバー・info.jsonは出力フォルダ直下にある

3. 既定ブラウザ（可能ならChrome）で開く（OS非依存のヘルパー。Chrome拡張のnavigateは`file://`不可なのでこれを使う）：


### 発信プロフィールの作成提案（STEP 3.5でプロフィールが無かった場合のみ）

活かし方を汎用で書いたときは、最後にこう投げかける：

> 「あなたの発信情報（誰に・何を発信しているか）が分かるもの（コンセプト設計の結果など）はありますか？あれば次回から『活かし方』をあなた向けに具体化できます。無ければ、3つほどの質問で"発信プロフィール"を作って保存できます。作りますか？（次回以降ずっと使えます）」

- **この会話で既にユーザーが発信情報を話していたら**、同じ質問を繰り返さず、それを使って「この内容で発信プロフィールに残しますか？」と確認して保存する
- ユーザーが「作る」と言ったら、簡単なヒアリング（誰に／何を／どう独自に／アカウント名・@／発信の目的／トーン・キャラ／実績・数字）をして、下記テンプレで `~/Documents/Claude/Projects/発信プロフィール/profile.md` に保存する（フォルダが無ければ作る）
- 保存したら「次回の分析から、活かし方があなたの発信に合わせて具体化されます」と伝える。押しつけない（「今はいい」ならそのまま終了）

## 発信プロフィール（profile.md）のテンプレ

保存先: `~/Documents/Claude/Projects/発信プロフィール/profile.md`（**1ユーザー1ファイル**。bunseki-reel だけでなく daihon-copy-make・競合分析でも共通に使える発信者の基本情報）。

## 注意事項

- ダウンロード・スクロール間隔を詰めない（IGのレートリミット・BAN回避）
- 収集した動画・画像は研究目的のローカル保存。転載しないようユーザーにも注意（報告書フッターで足りる）
- 途中で何かが失敗しても**報告書を最後まで届けることを最優先**。失敗は隠さず報告書とチャットの両方に書く
- リールに音声ナレーションが**本当に**無い（BGM＋テロップのみ）場合はフレームのテロップが「台本」になる。ただし**それを断定する前に必ずフレームで口の動きを確認する**（STEP 1の最重要注意）。文字起こしが空＝ナレーション無し、と短絡しないこと
