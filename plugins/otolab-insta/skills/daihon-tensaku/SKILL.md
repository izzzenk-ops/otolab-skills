# 台本辛口添削（daihon-tensaku）｜撮る前の台本を1行ずつ厳しく直す

## このスキルについて

ユーザーが**台本テキスト**を貼り付けると、動画を一切使わず**台本の言葉そのもの**を固定基準で厳しく採点・添削し、1枚のHTML報告書としてChromeで開く。核心は**台本を1フレーズずつ解説する**こと——各行で「視聴者に何が起きるか（感情の動き）／どう直すか／加えるといいワード」を突きつける。撮影前に言葉だけで直せるのが最大の価値。

**使い分け（重要）:**
- このスキル → **できあがった台本テキスト**を撮影前に添削・採点・1行解説
- `tensaku-reel` → **動画そのもの**（投稿済みリールのURL/動画ファイル）をテロップ・音・カバー込みで添削
- `daihon-copy-make` → 参考台本の型を移植して**新しい台本を作る**
- `daihon-*` 系（映画風・ギャル風などの専用台本スキル） → 台本を**ゼロから書く**

**入力は2パターン**（どちらでも受ける）:
1. **台本テキストの貼り付け**（チャットに直接）← 基本
2. **台本ファイルのパス**（.txt / .md / .docx）

このスキルは辛口分析くん（tensaku-reel）の思想と口調・デザインを流用した**軽量版**。動画パイプライン（yt-dlp・文字起こし・フレーム・音量実測）は一切使わないので、環境セットアップもコストも不要（Claude Code内で完結）。

## 出力先

```
~/Documents/Claude/Projects/台本添削/<タイトル>_<YYYYMMDD>/
├── script.txt      添削対象の元台本（貼り付けをそのまま保存）
└── report.html     添削報告書
```

`<タイトル>` は台本の内容から短く付ける（ファイル入力ならファイル名の幹）。同名なら連番。

## STEP 1：台本の取り込み

- **貼り付けテキスト** → そのまま `script.txt` に保存する（Writeツール。改変しない）
- **ファイルパス**（.txt/.md） → Readで読む。.docxなら `anthropic-skills:docx` で本文を抽出してから `script.txt` に保存
- 台本が短すぎる（1〜2行だけ）／台本ではなく企画メモの場合は、添削に入る前に「これは完成台本ですか？（フックからCTAまで通しの台本があると1行ずつ添削できます）」と一度だけ確認する

台本以外の付帯情報（テロップ指示・画の指示・秒数メモ）が混ざっていたら、**どれが読み上げ本文でどれが指示か**を切り分けてから採点する（指示は演出として評価材料に使い、台本本文の減点対象にはしない）。

## STEP 2：形式判定＋分解

1. **形式を判定**：ナレーション台本（読み上げ前提）か、テロップ台本（画面文字前提・喋りなし）か、両方か。形式でフックやリズムの正解が変わる
2. **1フレーズ＝1行に分解**：句点・意味の切れ目で区切って番号を振る。これが「1行ずつ解説」の単位になる（元の改行が粗ければこちらで適切な粒度に割り直す）
3. **文字数・想定尺を出す**：文字数をカウント。想定尺 ≒ 文字数 ÷ 5〜6字/秒（ナレーション台本のみ。テロップ台本は尺は「—」）

## STEP 3：発信プロフィールの確認（添削の個別化）

「誰に向けた発信か」でフックやCTAの正解が変わるので、採点前に確認する。**個別化の材料は「明示的なプロフィール」か「この会話でユーザーが話した情報」だけ。メモリを勝手にあさらない**（受講生配布で他人の情報が混ざるのを防ぐ）：

1. `~/Documents/Claude/Projects/発信プロフィール/profile.md` があれば読む（bunseki系・tensaku-reelと共通ファイル）
2. 無ければこの会話でユーザーが話した発信情報を使う
3. どちらも無ければ「①誰に向けた発信か ②この台本のゴール（保存/フォロー/コメント/販売）」だけ短く聞いてから採点する。答えたがらなければ汎用基準で進める

## STEP 4：視聴者心理の設計を先に読む

採点・添削の前に、台本全体を**視聴者に起きる効果**のレンズで読む（[[reference-viewer-psychology-blueprint]]）。**制作者の内心は断定しない。断定してよいのは視聴者側の反応まで。**

- **感情の弧**を掴む：どこで引き（好奇心）→緊張・期待→共感→納得・解放→行動、と動くか。山がどこにあるか、平坦な区間はどこか
- **伏線→回収**を追う：どこで疑問（振り）を植え、どこで回収するか。**回収されない振り**があれば欠陥として拾う
- 各フレーズが担う感情フェーズ（引き/緊張/共感/解放/CTA）を仮に割り当てておく（STEP 6のチップに使う）

**具体性ルール（最重要）**：感情の解説は抽象語で止めない。必ず**視聴者の頭に浮かぶ実際の言葉「〜」（主語・目的語入り）＋次に何をさせるか**まで書く（❌「疑問を仕込む」→ ✅「"なんでこの人、いきなり謝ってるの？"と思わせる→理由を知りたくて見続ける」）。

## STEP 5：採点（固定ルーブリック・100点満点）

**台本テキストに書かれていることだけを根拠にする。書かれていないもの（テロップの見やすさ・声・カバー）は減点しない**（それは動画の話＝tensaku-reelの担当）。以下の配点・基準は変えない：

| 軸 | 配点 | 満点の基準 | 大幅減点の例 |
|---|---|---|---|
| ① 冒頭フック | 25 | 最初の1〜2文（0〜2秒相当）に結論・違和感・数字・強い感情のどれかがあり、ターゲットが「自分の話だ」と分かる | 挨拶・名乗り・「今日は〜」で開始（〜12点）／2文すぎても本題が始まらない（〜15点） |
| ② 1メッセージ性 | 20 | 1本1メッセージに絞れている。視聴者が得るもの（メリット）が冒頭1/3までに明示 | メッセージが2つ以上混在（〜10点）／「見た人が得る物」が最後まで不明（〜8点） |
| ③ 構成・展開 | 20 | 中だるみなし。新情報が途切れず、転換が明確。伏線→回収がある | 同じ話の繰り返し／前置きが尺の20%超／回収されない振り（〜10点） |
| ④ 感情設計 | 15 | 視聴者の感情に山（引き→緊張→解放等）が設計され、どこで何を感じさせるか明確 | 平坦で山がない・感情の起伏ゼロ（〜7点）／共感も驚きも無い説明の羅列 |
| ⑤ 言葉・リズム | 10 | 冗長・重複なし。1フレーズが短く読みやすい。話し言葉として自然・テンポがある | 一文が長すぎて息継ぎできない／書き言葉で固い／同じ語尾の連続（〜5点） |
| ⑥ CTA・締め | 10 | 保存/フォロー/コメントの理由を作ってから誘導。締めに余韻か次への引きがある | CTAなし（〜4点）／「参考になったら嬉しいです」等の理由なし誘導（〜6点）／尻切れ |

- 数値基準はすべて**目安**。機械的に当てず、台本の実態を読んで判定する。ただし**甘くする方向に曲げない**
- 判定バンド: **S=90+（そのまま撮れる設計）／A=80+／B=70+／C=60+（撮れるが伸びにくい）／D=60未満（構成から作り直し推奨）**。よくある「普通に書いた台本」が45〜60点台に落ちるのは設計どおり。満点を安売りしない

**辛口キャップ（甘さ防止・必ず最後に上限チェック。迷ったら低い方に倒す）:**
- **軸キャップ**: その軸の役割が果たせていない「証拠つき欠陥」が1つ → その軸は**配点の60%が上限**。2つ以上 → **40%が上限**。機能はしているが取りこぼしがある → **80%が上限**。磨き込み提案（欠陥ではない改善）はキャップ対象外
- **入口キャップ**: 冒頭フックに証拠つき欠陥がある場合、**総合は69点以下（C以下）**。入口で失った視聴者は中身の出来に関係なく戻らない。中身が良いことは指摘文とフォローで伝え、点では甘くしない
- **校正アンカー**: **80点以上（A/S）を付けてよいのは、証拠つき欠陥を1つも見つけられなかった台本だけ**。構成が良くても冒頭フックに欠陥＝60〜65。1メッセージが破綻（言いたいことが2つ以上）＝どんなに文章が上手くても65以下
- 適用したキャップは報告書の採点表の一言に反映する（例:「入口キャップ適用: 冒頭が名乗りのため総合はC止まり」）

## STEP 6：1行ずつ解説（このスキルの核心）

STEP 2で分解した**全フレーズ**に、順番に次の3点を付ける。行を飛ばさない（良い行も「🌱維持」として残す）：

1. **👀 視聴者に起きること（感情の動き）** — その行が視聴者に何を感じさせ、頭に何を浮かべさせ、次にどうさせるか。**具体性ルール（STEP 4）を厳守**：視聴者の実際の言葉「〜」＋意図動詞（〜と思わせる/疑問を抱かせる/したくさせる）＋次の行動。フェーズチップ（引き/緊張/共感/解放/CTA）を1つ選ぶ
2. **✍️ 改善案** — その行をどう直すか。Before→Afterの粒度で。**直す必要のない良い行はこのセルを出さない**（報告書テンプレの `lr-fix` を省略）
3. **➕ 加えるといいワード** — その行に足すと効く具体語・フレーズ（例:「実は」「たった3秒で」「逆に」等の接続・数字・呼びかけ）＋なぜ効くか一言。**無ければ出さない**（`lr-word` を省略）

**辛口の資格（tensaku-reelと共通）:** すべての指摘は ①台本の実際の言葉を引用 ②視聴者に起きること ③具体的な直し方 の3点セット。「フックが弱い」だけの抽象ダメ出しは書かない。

## STEP 6.5：口調（熱血コーチ）— このスキルの人格

**総評・トップ3・各指摘は「🔥指摘（厳しく）→🤝フォロー→直し方」の温度で。**

- **指摘は厳しく断言する。** オブラートに包まない・「〜かもしれません」で逃げない。「はい、この1行目で視聴者はもう指を動かします」レベルの直球。**視聴者に何が起きるか**まで必ず書く
- **フォローで必ず受け止める。**「これは最初みんなやります」「惜しい！あと1語で化けます」と normalize し、「直せば伸びる」方向で締める
- 語り口は「です・ます」＋熱血の呼びかけ（「ここ、見てください」「変えましょう！」）。読者＝台本を書いた本人に直接話しかける
- **禁止**: 人格への言及（「センスがない」）／嘲笑・突き放し／採点を口調で甘くすること（優しさはフォローで、点は基準どおり）
- 締めは必ず「トップ3を直せば化ける。撮るのが楽しみだ」系の前向き着地

## STEP 7：リライト台本

「私が直すならこうする」を書く。**同じネタ・同じ出来事のまま**、構成と言葉だけを組み直す（別の企画を提案しない）。維持ポイント（🌱）は残す。

- **ブロックごとの BEFORE（元の台本）⇄ AFTER（リライト）対比で書く**（報告書テンプレの `rw-block`）。BEFOREは元台本から該当箇所を引用し、各ブロックに「何を変えたか」の一言タグ、変更しないブロックは「維持」タグ（`rw-change keep`）
- **AFTER欄には常に台本本文そのものを書く。** 維持ブロックも「同じ」と省略せず、BEFOREと同じ本文を再掲する（**AFTER列だけ縦読みで完成台本になる**のが基準）。演出・画の補足は本文の後ろに**（括弧書き）**
- コピー用の全文は隠し要素 `#rewrite-script` にAFTER本文のみ（1フレーズ＝1行、セクション間は空行）で入れる

## STEP 8：HTML報告書の生成

1. `<skill>/assets/report_template.html` を読み、**CSSと構造はそのまま**に `[[...]]` を実データで置換して `report.html` として保存
   - 採点リングの `style="--p:[[総合点]]"` と レーダーの `data-scores`（6軸）を忘れず数値に置換（`data-max="25,20,20,15,10,10"` は固定）
   - 6軸の採点バーの width は `得点/配点*100`%
   - **1行ずつ解説（`line-row`）を台本の全フレーズ分複製**。フェーズチップは `phase-hook`/`phase-tension`/`phase-empathy`/`phase-payoff`/`phase-cta` を使い分け。良い行は `class="line-row good"` にして `lr-fix`/`lr-word` を省略
   - 感情の弧・伏線→回収を §3 に。リライトの `rw-block` を §6 に。元台本全文を `tr-text` にHTMLエスケープして埋め込み
   - ⚠️ **本文に内部・技術用語を出さない**（segments・dB・スキル名の生出し等は禁止。平易に言い換える）
2. フッターに「台本テキストのみの評価」である旨は既にテンプレに固定済み。確認できなかった項目があれば正直に追記
3. 開く：

```bash
python3 <skill>/scripts/open_report.py "<out_dir>/report.html"   # Mac/Linux
python  <skill>\scripts\open_report.py "<out_dir>\report.html"   # Windows
```

可能ならヘッドレスChromeでスクショして崩れ（特に**レーダーの6角形が描画されているか**）を確認する：
```bash
"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu \
  --screenshot=<scratchpad>/report_check.png --window-size=1200,3200 --hide-scrollbars \
  "file://<out_dir>/report.html"
```
6角形が出ていなければ `data-scores`/`data-max` の置換ミスを疑う。

## STEP 9：完了報告

- **総合点と判定を最初に言い切る**（例:「62点・C判定。冒頭の1文を変えれば化けます」）
- 最優先トップ3を1行ずつ＋報告書のフルパス
- 熱血コーチの温度で締める（厳しさの後に「直せば伸びる」）
- 「直したらもう一度貼ってください。再添削します」と添える
- 発信プロフィールが無かった場合は tensaku-reel と同じ流儀で作成を提案してよい（`~/Documents/Claude/Projects/発信プロフィール/profile.md`・押しつけない）

## 注意事項

- 採点・指摘は**この台本に書かれている言葉**から。一般論だけの減点（「普通は〜だから」）をしない
- 動画としての出来（テロップ・音・カバー・実際の秒配分）は評価対象外。ユーザーが動画の添削も欲しがったら tensaku-reel を案内する
- 参考として貼られた**他人の台本**を「これを添削して」と持ち込まれた場合もスキルは動くが、「自分の台本を撮影前に直す」用途が本領なので一言添える
- 途中で何かが失敗しても**報告書を最後まで届けることを最優先**。失敗は隠さず報告書とチャットの両方に書く
